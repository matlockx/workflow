## Content Rules & Hints (to get your ideal tone)

- **Problem:** numbered list (1..N) of precise failure modes; quote exact DB/stack errors from logs if present.
- **Root Cause:** strictly technical; show **where** in code (file + function/class) and **why** it failed (schema mismatch, upsert vs update semantics, loop patterns, race conditions, etc.).
- **Solution:** one subsection per commit with:

  - **Problem**
  - **Fix**
  - **Result**
  - (Optional) a short code diff or SQL example if it clarifies.
- **Cumulative Changes:** include the raw `--stat` block in a fenced code section, then a tidy “Key Changes by File” bullet list with **(+/- line counts)** and micro-summaries.
- **Testing:** list the scenarios; separate “existing updated” vs “newly added”. Mention edge cases.
- **Safety & Compatibility:** bullets for transactions, flags, API stability, schema usage.
- **Performance Impact:** show **Before/After** query counts/latency with a compact table; add a quick “Query Breakdown” mini-block.
- **Success Criteria:** use checkboxes `[x]` exactly as in your sample.
- **Evolution:** explain why multiple commits were needed and how each de-risked the next.
- **Follow-up Work:** concrete, non-blocking tasks.
- **Related:** ticket, pattern references, schema docs (file paths + lines if known).
